# TDD Rule

**이 규칙은 Rigid — 상황에 맞게 변형하지 마라.**

---

## 프로젝트별 규칙 우선 조항

이 파일은 **기본값(fallback)** 이다. 프로젝트의 `CLAUDE.md` 또는 `AGENTS.md`에 TDD 관련 규칙이 명시되어 있으면 **그 규칙이 이 파일보다 우선**한다.

---

## 적용 기준

**TDD 필수 적용 대상:**
- 비즈니스 로직 (계산, 변환, 검증)
- API 엔드포인트 핸들러
- 유틸리티 함수
- 버그 수정 — 재현 테스트를 먼저 작성한 뒤 수정

**TDD 불필요 대상:**
- 타입 정의 파일 (`*.d.ts`, `types/index.ts` 등)
- 설정 파일 (`vite.config.ts`, `eslint.config.js`, `tsconfig.json` 등)
- 순수 UI 컴포넌트 (로직 없이 JSX만 반환하는 컴포넌트)
- SQL 마이그레이션 스크립트
- 상수 선언

## RED-GREEN-REFACTOR 사이클

### RED

1. **하나의 동작 = 하나의 테스트.** 여러 동작을 한 테스트에 묶지 마라.
2. 테스트를 작성한 뒤 **반드시 실행해서 실패를 눈으로 확인**하라.
3. 실패 이유가 반드시 **"기능 미구현"** 이어야 한다. 컴파일 에러나 import 오류로 실패하는 것은 RED가 아니다 — 그 오류를 먼저 해결하라.

### GREEN

1. 테스트를 통과시키기 위한 **최소한의 코드만** 작성한다.
2. **YAGNI** — 지금 테스트가 요구하지 않는 기능을 미리 구현하지 마라.
3. 신규 테스트뿐 아니라 **기존 테스트 전체가 통과하는지 확인**하라.

### REFACTOR

1. **중복 제거, 이름 개선, 헬퍼 추출**만 허용한다.
2. 리팩터링 중 테스트가 깨지면 즉시 되돌려라 — green을 유지하는 것이 최우선이다.
3. **새로운 동작을 추가하지 마라.** 동작 추가는 다음 RED 단계에서 시작한다.

### 반복

리팩터링이 끝나면 **다음 동작에 대한 RED로 돌아간다.**

---

## 삭제 강제 규칙

- 테스트 전에 프로덕션 코드를 먼저 작성했다면 **즉시 삭제하고 RED부터 재시작**하라.
- "참고용으로만 남기겠다"는 **금지**다. 참고는 git history로 한다.

## 변명 차단표

| 변명 | 반론 |
|------|------|
| "너무 단순해서 테스트가 불필요하다" | 단순한 코드일수록 테스트 작성 비용이 낮다. 단순함은 면제 사유가 아니다. |
| "나중에 테스트를 추가하겠다" | 나중은 오지 않는다. 동작하는 코드에 테스트를 끼워 넣는 것은 TDD가 아니다. |
| "시간이 없다" | TDD는 디버깅 시간을 줄인다. 지금 아끼는 10분이 나중에 몇 시간이 된다. |
| "코드를 삭제하면 낭비다" | 잘못된 순서로 작성된 코드를 유지하는 것이 낭비다. git stash로 보존하고 삭제하라. |
| "프로토타입이라 TDD가 맞지 않는다" | 프로토타입 코드가 프로덕션에 남는다. 처음부터 올바른 습관을 유지하라. |
